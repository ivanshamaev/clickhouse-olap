Note ClickHouse and the Rust engine can share the same VPS. ClickHouse listens only on 127.0.0.1:8123 by default — the engine connects locally, nginx handles external HTTPS traffic.

Install ClickHouse

Add the official ClickHouse apt repository and install:

# Update the system first
apt update && apt upgrade -y

# Install prerequisites
apt install -y apt-transport-https ca-certificates curl gnupg

# Add ClickHouse signing key
curl -fsSL https://packages.clickhouse.com/rpm/lts/repodata/repomd.xml.key \
  | gpg --dearmor -o /usr/share/keyrings/clickhouse-keyring.gpg

# Add repository
echo "deb [signed-by=/usr/share/keyrings/clickhouse-keyring.gpg] \
  https://packages.clickhouse.com/deb stable main" \
  > /etc/apt/sources.list.d/clickhouse.list

# Install
apt update && apt install -y clickhouse-server clickhouse-client

# Enable and start
systemctl enable --now clickhouse-server
systemctl status clickhouse-server

Verify ClickHouse is listening on the loopback interface only:

ss -tlnp | grep 8123
# expected: 127.0.0.1:8123

Create database & user

Create an isolated database and a dedicated application user with minimal privileges:

clickhouse-client --query "
  CREATE DATABASE IF NOT EXISTS olap;

  CREATE USER IF NOT EXISTS olap_user
    IDENTIFIED BY 'REPLACE_WITH_STRONG_PASSWORD';

  GRANT ALL ON olap.* TO olap_user;
"
Security Replace REPLACE_WITH_STRONG_PASSWORD with a randomly generated secret (e.g. openssl rand -base64 32). Never use the default ClickHouse user for the application.

Table schema

The example schema models a sales dataset. Adjust column names and types to match your data. The engine reads table and field names from the semantic model TOML file — the schema here just needs to match. 

clickhouse-client --user olap_user --password 'YOUR_PASSWORD' --query "
CREATE TABLE IF NOT EXISTS olap.sales (
    order_id         UInt64,
    customer_id      UInt64,
    order_date       Date,
    region           LowCardinality(String),
    country          LowCardinality(String),
    product_category LowCardinality(String),
    amount           Decimal(18, 2),
    qty              UInt32
) ENGINE = MergeTree()
ORDER BY (region, order_date, order_id)
SETTINGS index_granularity = 8192;
"

Schema design tips

  • Use LowCardinality(String) for dimension columns with < 10 000 distinct values (region, category, status, …). It compresses 5–10× better and speeds up GROUP BY.
  • Choose the ORDER BY key to match your most common pivot dimensions. ClickHouse uses it as the primary index for range scans.
  • Use Date (not DateTime) for date dimensions when time-of-day precision is not needed — it halves storage and speeds up toStartOfQuarter()-style expressions.
  • Partitioning by month (PARTITION BY toYYYYMM(order_date)) helps DROP old data and speeds date-range queries, but is not required for the engine to function.

For production, load your real data via CSV import:

# Import from a CSV file (no headers)
clickhouse-client --user olap_user --password 'YOUR_PASSWORD' \
  --query "INSERT INTO olap.sales FORMAT CSV" < data.csv

# Or with headers (CSVWithNames)
clickhouse-client --user olap_user --password 'YOUR_PASSWORD' \
  --query "INSERT INTO olap.sales FORMAT CSVWithNames" < data_with_headers.csv

Verify

# Row count
clickhouse-client --user olap_user --password 'YOUR_PASSWORD' \
  --query "SELECT count() FROM olap.sales"

# Quick aggregation test
clickhouse-client --user olap_user --password 'YOUR_PASSWORD' \
  --query "
  SELECT region, sum(amount) AS revenue, uniq(customer_id) AS customers
  FROM olap.sales
  GROUP BY region
  ORDER BY revenue DESC
  FORMAT PrettyCompact
"

Security notes

  • ClickHouse must listen only on 127.0.0.1:8123 — never expose port 8123 to the internet.
  • Use a dedicated user (olap_user) with GRANT ON olap.* only. No system-wide grants.
  • ClickHouse has no TLS on the HTTP interface by default — that's fine because traffic never leaves localhost. External clients go through nginx HTTPS.
  • Rotate the olap_user password periodically and update config.toml on the engine side.
