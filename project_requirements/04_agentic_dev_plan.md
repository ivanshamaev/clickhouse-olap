# Документ 04. План агентной разработки с Claude Code

**Проект:** OLAP-анализ данных ClickHouse в Excel через Rust-движок
**Статус:** Черновик v0.1
**Связанные документы:** `00`–`03` (требования и архитектура)

---

## 1. Идея подхода

Разработка ведётся итеративно с помощью **Claude Code** как агента-исполнителя.
Человек выступает в роли архитектора/ревьюера: задаёт фазу, проверяет результат,
утверждает переход к следующей. Claude Code работает в репозитории, где заранее
настроено окружение:

- **`CLAUDE.md`** — постоянный контекст проекта (грузится в каждую сессию): архитектурные инварианты, стек, команды сборки/тестов, правила.
- **Skills** (`.claude/skills/<name>/SKILL.md`) — узкоспециализированные инструкции, подгружаемые по релевантности: генерация SQL к ClickHouse, формовка pivot, контракт API, тестирование.
- **Subagents** (`.claude/agents/<name>.md`) — изолированные исполнители для задач, которые не должны засорять основной контекст: ревью кода, исследование, проверка корректности итогов.
- **`docs/`** — требования (`00`–`03`) как источник истины, на который ссылается агент.

Принцип: **архитектурные инварианты (D1–D5) и высокорисковые правила (R2/R5) живут в окружении агента**, чтобы он не «забывал» их между сессиями и не предлагал решений, нарушающих их (например, агрегацию сырья в Rust).

## 2. Инварианты, которые агент НЕ должен нарушать

Эти правила выносятся в `CLAUDE.md` и дублируются в релевантных skills:

1. **Агрегация — только в ClickHouse** (D1). Rust не агрегирует сырьё. Любой код, тянущий сырые строки для агрегации в Rust, — ошибка.
2. **Rust-движок stateless** (FR-ENGINE-01). Никакого пользовательского состояния между запросами, кроме кэша.
3. **Только поля/таблицы семантической модели** (FR-QUERY-04). Никакого произвольного SQL от клиента.
4. **Параметризация SQL** (FR-QUERY-03). Никакой конкатенации недоверенных значений.
5. **Контроль кардинальности до запроса** (FR-QUERY-05, R2). Запрос, грозящий некомпактным предагрегатом, отвергается до исполнения.
6. **Неаддитивные итоги — из ClickHouse** (FR-PIVOT-04b, R5). Никогда не суммировать подытоги для count distinct / перцентилей / avg.
7. **Контракт версионируем и стабилен** (NFR-COMPAT-03). Изменения контракта — осознанные, с версией.
8. **Тонкий клиент** (D4). Никаких вычислений в надстройке.

## 3. Фазы разработки

Каждая фаза — это законченный, тестируемый инкремент. Переход к следующей — после
прохождения тестов и ревью человеком.

### Фаза 0. Каркас окружения и проекта
**Цель:** настроить окружение Claude Code и скелет монорепозитория.
- Репозиторий со структурой: `engine/` (Rust), `addin/` (Office.js), `docs/`, `.claude/`.
- `CLAUDE.md`, базовые skills и subagents (см. промпт ниже).
- CI: сборка, линт, тесты (заглушки).
- Контракт (doc 03) зафиксирован как JSON-схема в `docs/contract/`.
**Критерий готовности:** `cargo build` и сборка надстройки проходят на пустом скелете; CI зелёный.

### Фаза 1. Контракт и семантическая модель
**Цель:** формализовать контракт и модель, без обращения к реальному ClickHouse.
- JSON-схемы запроса/ответа/метаданных/ошибок (раздел 4–7 doc 03).
- Парсер и валидатор семантической модели (FR-SEM-01..07).
- Валидация входного запроса по схеме и модели (FR-QUERY-04, NFR-SEC-07).
- Юнит-тесты валидации, в т.ч. negative-кейсы (поле вне модели → `FIELD_NOT_ALLOWED`).
**Критерий:** валидатор отвергает некорректные запросы, принимает корректные; golden-тесты схемы.

### Фаза 2. Генерация агрегирующего SQL
**Цель:** структурированный запрос → агрегирующий SQL к ClickHouse (без итогов).
- Планировщик: GROUP BY, фильтры, сортировки, Top-N — всё в SQL (FR-QUERY-02).
- Параметризация значений (FR-QUERY-03).
- Контроль кардинальности (FR-QUERY-05, R2): оценка числа групп, `PREAGGREGATE_TOO_LARGE`.
- Golden SQL-тесты (NFR-MAINT-02): эталоны сгенерированного SQL.
**Критерий:** для набора эталонных запросов SQL совпадает с golden; high-cardinality запросы отвергаются.

### Фаза 3. Исполнение против ClickHouse
**Цель:** реальное обращение к тестовому ClickHouse.
- Пул соединений, таймауты, ретраи транзиентных ошибок, отмена (FR-ENGINE-02..06).
- Лимит объёма принимаемого предагрегата (FR-ENGINE-07).
- Интеграционные тесты против ClickHouse в Docker с тестовым датасетом.
**Критерий:** запросы исполняются, возвращают предагрегат; таймаут/отмена/ретрай работают.

### Фаза 4. Формовка pivot и итоги
**Цель:** укладка предагрегата в pivot + корректные итоги. **Самая рискованная фаза.**
- Многоуровневые оси rows/columns, ячейки (FR-PIVOT-01).
- Аддитивные итоги в Rust (FR-PIVOT-04a).
- **Неаддитивные итоги из ClickHouse** (FR-PIVOT-04b, R5) — отдельные запросы/ROLLUP по уровням.
- Иерархии дат (FR-PIVOT-05), вычисляемые меры (FR-PIVOT-06).
- Тесты корректности итогов: сверка count distinct/перцентилей с эталоном, проверка, что итог ≠ сумме подытогов там, где это так.
**Критерий:** итоги по неаддитивным мерам корректны на эталонном датасете; формовка детерминирована (FR-PIVOT-08).

### Фаза 5. Кэш и режимы деплоя
**Цель:** кэширование + два режима запуска.
- Кэш по ключу (FR-CACHE-01), TTL/инвалидация (FR-CACHE-03).
- Общий внешний кэш для серверного режима (FR-CACHE-02) с учётом прав (FR-CACHE-04, NFR-SEC-08).
- Локальный и серверный режимы из одной кодовой базы (FR-DEPLOY-01..03).
**Критерий:** кэш-хиты ускоряют повторные запросы; ключ учитывает контекст доступа; оба режима работают.

### Фаза 6. Тонкий клиент Office.js (MVP UI)
**Цель:** надстройка под MS Office: конструктор + отрисовка.
- Подключение/health-check, выбор модели, конструктор pivot (FR-ADDIN-01..11).
- Отправка запроса, индикатор/отмена, отрисовка результата (FR-ADDIN-12..16).
- Проверка ограничений Office.js на нативный pivot (R1) — нативная сводная или форматированный диапазон.
**Критерий:** end-to-end сценарий «собрал pivot в Excel → получил корректную таблицу» на тестовом ClickHouse.

### Фаза 7. Наблюдаемость, безопасность, нагрузочное
**Цель:** довести до эксплуатационной готовности.
- Логи с `request_id`, метрики (FR-MON), маскирование (NFR-SEC-06).
- TLS/аутентификация серверного режима (NFR-SEC-03/05), решение по R4.
- Нагрузочное тестирование под целевой масштаб (NFR-SCALE-02), drill-down латентность.
**Критерий:** целевые перцентили латентности достигнуты; нагрузочный профиль под сотни пользователей выдержан.

## 4. Рабочий цикл одной агентной итерации

1. **Человек:** «Реализуй задачу X из Фазы N» со ссылкой на конкретные FR/NFR.
2. **Claude Code:** читает `docs/`, релевантные skills и `CLAUDE.md`; планирует; реализует; пишет тесты.
3. **Subagent-ревьюер:** проверяет инварианты (раздел 2) и качество, возвращает summary.
4. **Claude Code:** правит по замечаниям; гоняет тесты/линт.
5. **Человек:** ревью diff, утверждение, мердж. `/clear` перед новой несвязанной задачей.

Рекомендации по гигиене контекста: использовать subagents для исследования и ревью
(изоляция контекста), `/clear` между несвязанными задачами, держать высокорисковые
правила в `CLAUDE.md`, чтобы они не терялись при компактификации.

## 5. Какие skills и subagents нужны (реализуются в Фазе 0)

**Skills** (`.claude/skills/`):
- `clickhouse-sql` — как генерировать корректный/безопасный агрегирующий SQL под нашу версию ClickHouse, контроль кардинальности, ROLLUP/GROUPING SETS для итогов.
- `pivot-shaping` — алгоритмы укладки предагрегата в оси/ячейки, правила аддитивных vs неаддитивных итогов.
- `api-contract` — структура запроса/ответа/ошибок, правила версионирования.
- `rust-engine-conventions` — стиль кода, обработка ошибок, async/пул, stateless-правила.
- `officejs-addin` — ограничения Office.js, отрисовка pivot, тонкий клиент.
- `testing` — golden SQL, юнит/интеграционные тесты, тесты корректности итогов.

**Subagents** (`.claude/agents/`):
- `code-reviewer` — ревью на инварианты раздела 2 и качество (изолированный контекст).
- `invariant-checker` — узкий проверяющий: не нарушены ли D1/R2/R5 в diff.
- `researcher` — исследование (например, поведение ClickHouse-функций) без засорения основного контекста.

## 6. Риски агентной разработки и их снижение

| Риск | Снижение |
|------|----------|
| Агент предложит агрегировать сырьё в Rust (нарушение D1) | Инвариант №1 в `CLAUDE.md` + `invariant-checker` subagent |
| Неверные неаддитивные итоги (R5) | Skill `pivot-shaping` + обязательные тесты корректности в Фазе 4 |
| Дрейф контракта между клиентом и движком | Контракт как single source of truth в `docs/contract/`, golden-тесты схемы |
| Потеря инвариантов при компактификации контекста | Compact-инструкции в `CLAUDE.md`, ключевые правила продублированы в skills |
| Засорение контекста длинными исследованиями | Делегирование в subagents с изолированным контекстом |
| Небезопасный SQL (инъекции) | Инвариант №4 + skill `clickhouse-sql` + тесты negative-кейсов |

## 7. Что подтвердить до старта Фазы 0

- Версия ClickHouse и доступ к тестовому инстансу (Docker) — для Фаз 2–4.
- Язык/тулчейн Office.js-надстройки (TypeScript предполагается) и способ локального запуска.
- Решение по аутентификации до ClickHouse (R4) — желательно до Фазы 5.
- Конкретные пороги кардинальности (`max_groups`) под реальные модели (R2).
