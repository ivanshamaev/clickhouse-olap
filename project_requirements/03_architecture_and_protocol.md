# Документ 03. Архитектура и контракт обмена

**Проект:** OLAP-анализ данных ClickHouse в Excel через Rust-движок
**Статус:** Черновик v0.2
**Связанные документы:** `00_overview_vision.md`, `01_functional_requirements.md`, `02_nonfunctional_requirements.md`

---

## 1. Назначение

Описывает компонентную архитектуру и **контракт обмена** «надстройка ↔ движок».
Контракт — центральный стабильный артефакт, версионируемый (NFR-COMPAT-03).
JSON-структуры ниже — проектное предложение формы, уточняется на детальном проектировании.

Опорные решения: тяжёлая агрегация в ClickHouse (D1); Rust — stateless-формовщик
компактного предагрегата (D2); два режима деплоя из одной кодовой базы (D3);
тонкий Office.js-клиент только под MS Office (D4/D5).

### 2.1. Ответственность компонентов

| Компонент | Ответственность |
|-----------|-----------------|
| Надстройка (Office.js) | Конструктор запроса + отрисовка результата. Без вычислений. |
| API-слой | Приём, версии API, `request_id`, аутентификация (серверный режим) |
| Валидация + контроль кардинальности | Проверка по схеме и модели; отклонение запросов, грозящих некомпактным предагрегатом (FR-QUERY-05, R2) |
| Планировщик | Структурированный запрос → **агрегирующий** SQL к ClickHouse; план уровней итогов |
| Исполнитель | Пул, параллелизм, таймауты, ретраи, отмена |
| Формовка | Укладка компактного предагрегата в pivot: оси, ячейки, итоги, вычисляемые меры, форматы |
| Кэш | Lookup/store; в серверном режиме — внешний общий, с учётом прав (R4) |
| Семантическая модель | Источник истины о таблицах/полях/мерах/агрегациях/лимитах |

## 3. Транспорт и режимы (D3)

- **Локальный режим:** движок слушает loopback (`127.0.0.1:<port>`), надстройка обращается по HTTP/JSON. TLS опционален (NFR-SEC-05). Для лёгких преобразований одного пользователя.
- **Серверный/кластерный режим:** 1..N stateless-экземпляров за балансировщиком, TLS обязателен, общий внешний кэш, аутентификация запросов. Для большого многопользовательского масштаба.
- **gRPC** — опциональная альтернатива HTTP/JSON; контракт сообщений эквивалентен.

Контракт **идентичен** в обоих режимах; различается только endpoint и параметры
безопасности. Версионирование: `/v1/...`, несовместимые изменения → `/v2`.

### 3.1. Операции (черновик)

| Метод | Назначение | FR |
|-------|-----------|----|

## 4. Контракт: запрос pivot (`POST /v1/query`)

> Описывает намерение пользователя, не SQL. Иллюстративная форма.

```json
{
  "api_version": "1",
  "request_id": "client-generated-uuid",
  "model_id": "sales_model",
  "rows": [
    { "field": "region", "sort": { "by": "measure", "measure": "revenue", "dir": "desc" }, "top_n": 20 },
    { "field": "product_category" }
  ],
  "columns": [
    { "field": "order_date", "date_granularity": "month" }
  ],
  "measures": [
    { "id": "revenue", "field": "amount",  "agg": "sum" },
    { "id": "orders",  "field": "order_id", "agg": "count_distinct" },
    { "id": "avg_check", "type": "calculated", "expr": "revenue / orders" }
  ],
  "filters": [
    { "field": "order_date", "op": "between", "value": ["2025-01-01", "2025-12-31"] },
    { "field": "country", "op": "in", "value": ["LV", "LT", "EE"] }
  ],
  "totals": { "row_subtotals": true, "column_subtotals": true, "grand_total": true },
  "limits": { "max_cells": 500000, "max_groups": 200000, "bypass_cache": false }
}
```

Замечания:
- `agg` ограничен разрешёнными моделью (FR-SEM-02); все агрегации идут в ClickHouse (D1).
- `type: calculated` — вычисляется в Rust ПОСЛЕ агрегатов (FR-PIVOT-06).
- `top_n` / `sort by measure` транслируются в ClickHouse (FR-QUERY-02).
- `limits.max_cells` — защита Excel (FR-RESULT-03); `limits.max_groups` — защита компактности предагрегата (FR-QUERY-05/R2). Превышение → управляемая ошибка ДО тяжёлого запроса.

## 5. Контракт: ответ pivot

```json
{
  "api_version": "1",
  "request_id": "client-generated-uuid",
  "status": "ok",
  "from_cache": false,
  "data_version": "2026-05-23T08:00:00Z",
  "row_axis": {
    "fields": ["region", "product_category"],
    "members": [
      { "key": ["LV", "Electronics"] },
      { "key": ["LV", "__TOTAL__"], "is_subtotal": true },
      { "key": ["__GRAND_TOTAL__"], "is_grand_total": true }
    ]
  },
  "column_axis": {
    "fields": ["order_date@month"],
    "members": [ { "key": ["2025-01"] }, { "key": ["2025-02"] } ]
  },
  "measures": [
    { "id": "revenue", "format": "currency", "type": "number" },
    { "id": "orders",  "format": "integer",  "type": "number" }
  ],
  "cells": [
    { "r": 0, "c": 0, "m": "revenue", "v": 12500.50 },
    { "r": 0, "c": 0, "m": "orders",  "v": 87 }
  ],
  "warnings": [],
  "stats": { "subqueries": 3, "engine_ms": 45, "clickhouse_ms": 880, "preaggregate_rows": 4200 }
}
```

Замечания:
- Разреженный список `cells` экономит объём при пустых пересечениях; плотная матрица — альтернатива (решить на проектировании).
- `is_subtotal`/`is_grand_total` помогают форматировать и проверять корректность неаддитивных итогов (FR-PIVOT-04b).
- `stats.engine_ms` vs `clickhouse_ms` отражает D2: время движка мало, основное — ClickHouse.

## 6. Контракт: метаданные модели (`GET /v1/models/{id}`)

```json
{
  "model_id": "sales_model",
  "dimensions": [
    { "field": "region", "type": "categorical" },
    { "field": "order_date", "type": "date", "hierarchy": ["year", "quarter", "month", "day"] },
    { "field": "amount_bucket", "type": "numeric_bucket", "bucket_size": 100 }
  ],
  "measures": [
    { "id": "revenue", "field": "amount",   "allowed_agg": ["sum", "avg", "min", "max"], "additive": true },
    { "id": "orders",  "field": "order_id", "allowed_agg": ["count", "count_distinct"], "additive": false }
  ],
  "filterable_fields": ["region", "country", "order_date", "product_category"],
  "guards": { "max_groups": 200000, "high_cardinality_fields": ["order_id", "customer_id"] }
}
```

- `additive` управляет стратегией итогов (FR-PIVOT-04a/b).
- `guards` поддерживает контроль кардинальности (FR-SEM-04, FR-QUERY-05): запрет/предупреждение при GROUP BY по `high_cardinality_fields` без агрегации.

## 7. Ошибки

```json
{
  "status": "error",
  "request_id": "client-generated-uuid",
  "error": {
    "code": "RESULT_TOO_LARGE",
    "message_user": "Слишком большой результат. Сузьте фильтр или уберите измерение.",
    "retriable": false
  }
}
```

Черновой словарь: `VALIDATION_FAILED`, `FIELD_NOT_ALLOWED`, `RESULT_TOO_LARGE`,
`PREAGGREGATE_TOO_LARGE` (контроль компактности, R2), `QUERY_TIMEOUT`,
`CLICKHOUSE_UNAVAILABLE` (retriable), `CANCELLED`, `INTERNAL`.

## 8. Стратегия планирования (структурированный запрос → агрегирующий SQL)

1. **Вся агрегация — в ClickHouse** (D1, FR-QUERY-02): GROUP BY по измерениям, фильтры, агрегаты, Top-N, сортировки — в SQL.
2. **Контроль компактности ДО исполнения** (FR-QUERY-05, R2): оценка числа групп; при риске превышения `max_groups`/высокой кардинальности — `PREAGGREGATE_TOO_LARGE` без запуска тяжёлого запроса.
3. **Итоги по уровням** (FR-PIVOT-04): для аддитивных мер итоги МОГУТ собираться в Rust из частичных результатов; для неаддитивных (count distinct, перцентили, avg) — берутся из ClickHouse на нужном уровне (через `WITH ROLLUP`/`GROUPING SETS` или отдельные агрегирующие запросы), НЕ суммированием подытогов (R5).
4. **Параллельное исполнение** независимых подзапросов (FR-ENGINE-03) с лимитом параллелизма и общим лимитом на ClickHouse (NFR-SCALE-03).
5. **Формовка в Rust** (D2): укладка компактного предагрегата по осям, расчёт вычисляемых мер, форматы. Память мала, так как набор компактный.
6. **Кэш** (FR-CACHE): ключ = модель + измерения + меры + фильтры + `data_version` (+ контекст прав при пробросе identity, R4/NFR-SEC-08).

> Ключевая инженерная развилка: способ получения корректных итогов для неаддитивных
> мер (ROLLUP/GROUPING SETS vs отдельные запросы). Решается на проектировании по
> критериям корректности и нагрузки на ClickHouse.

## 9. Граничные случаи

| Случай | Поведение |
|--------|-----------|
| Запрос даёт некомпактный предагрегат (высокая кардинальность) | `PREAGGREGATE_TOO_LARGE` до исполнения (FR-QUERY-05) |
| Результат > лимит Excel | `RESULT_TOO_LARGE`, без обрезки (FR-RESULT-03) |
| Неаддитивная мера + итоги | Итоги из ClickHouse, не сумма подытогов (FR-PIVOT-04b) |
| Пустой срез | Валидный ответ с пустыми осями |
| Отмена долгого запроса | `CANCELLED`, освобождение соединений (FR-ENGINE-05) |
| Транзиентный сбой ClickHouse | Ретрай; при исчерпании — `CLICKHOUSE_UNAVAILABLE` (retriable) |
| Поле вне модели | `FIELD_NOT_ALLOWED` на валидации (NFR-SEC-07) |
| Общий кэш + разные права пользователей | Ключ кэша учитывает контекст доступа (NFR-SEC-08) |
| Деление на ноль в вычисляемой мере | Политика фиксируется (null/0/ошибка) |

## 10. К фиксации на детальном проектировании

1. Финальная JSON-схема запроса/ответа (или proto) + версия.
2. Механизм итогов per-тип меры (раздел 8, п.3) — главная развилка.
3. Плотный vs разреженный `cells`.
4. Политика кэша, `data_version` (как определять свежесть данных ClickHouse), учёт прав (R4).
5. Аутентификация и модель доступа (R4 / NFR-SEC-03/08).
6. Отрисовка в Excel при ограничениях Office.js на нативные pivot (R1).
7. Конкретные пороги `max_groups` / контроля кардинальности под реальные модели (R2).
