# Документ 05. Стартовый промпт для настройки окружения Claude Code

**Назначение:** этот документ содержит готовый к вставке промпт. Запустите Claude Code
в пустом (или почти пустом) репозитории проекта, скопируйте блок ниже целиком и
отправьте как первое сообщение. Claude Code создаст структуру окружения: `CLAUDE.md`,
skills, subagents и каркас `docs/`.

> Предполагается, что документы требований `00`–`04` уже лежат в `docs/` репозитория
> (или будут добавлены). Промпт ссылается на них как на источник истины.

---

## Промпт (скопировать целиком)

```text
Ты — агент-исполнитель для проекта "OLAP-анализ данных ClickHouse в Excel через Rust-движок".
Твоя ПЕРВАЯ задача — НЕ писать код приложения, а настроить окружение Claude Code для этого
репозитория. Создай файлы окружения по инструкции ниже. После создания каждого файла кратко
покажи его назначение. Не реализуй бизнес-логику на этом шаге.

КОНТЕКСТ АРХИТЕКТУРЫ (это инварианты, нарушать их нельзя):
1. Тяжёлая агрегация выполняется ТОЛЬКО в ClickHouse. Сырьё — сотни млн строк — НЕ покидает
   ClickHouse для агрегации. Rust НИКОГДА не агрегирует сырьё.
2. Rust-движок — это stateless "формовщик": принимает компактный ПРЕДАГРЕГАТ от ClickHouse
   и укладывает его в форму pivot-таблицы (оси строк/столбцов, ячейки, итоги) + лёгкая
   постобработка (вычисляемые меры, форматы). Низкая нагрузка на память.
3. Движок запускается в двух режимах из одной кодовой базы: локальный (на машине пользователя)
   и серверный/кластерный (большой многопользовательский масштаб). Контракт одинаков.
4. Клиент — тонкая Office.js-надстройка ТОЛЬКО под MS Excel (Windows/macOS/web). Клиент НЕ
   вычисляет — только формирует запрос и отрисовывает результат. LibreOffice/OpenOffice — вне
   текущей фазы.
5. Доступ — только к полям/таблицам семантической модели. Произвольный SQL от клиента запрещён.
   Все недоверенные значения параметризуются (защита от SQL-инъекций).
6. Контроль кардинальности ДО исполнения: запрос, грозящий некомпактным предагрегатом
   (например GROUP BY по высоко-кардинальному полю), отвергается до тяжёлого запроса.
7. Неаддитивные итоги (count distinct, перцентили, avg) берутся из ClickHouse на нужном уровне
   агрегации (ROLLUP/GROUPING SETS или отдельные запросы), НИКОГДА не суммированием подытогов.
8. Контракт API версионируем и стабилен; изменения контракта осознанные.

ИСТОЧНИК ИСТИНЫ: документы требований лежат в docs/ (файлы 00_*..04_*). Если их там нет —
сообщи об этом и не выдумывай требования. Всегда сверяйся с ними по идентификаторам FR-*/NFR-*.

СОЗДАЙ СЛЕДУЮЩУЮ СТРУКТУРУ:

A) CLAUDE.md в корне репозитория. Включи в него:
   - Краткое описание проекта и его цели (1 абзац).
   - Раздел "Архитектурные инварианты" — перечисли пункты 1–8 выше дословно по смыслу, как
     ЖЁСТКИЕ правила, которые нельзя нарушать ни в одном предложении кода или дизайна.
   - Раздел "Структура репозитория": engine/ (Rust-движок), addin/ (Office.js-надстройка),
     docs/ (требования), .claude/ (окружение агента).
   - Раздел "Команды": плейсхолдеры для сборки/тестов/линта движка (cargo build, cargo test,
     cargo clippy) и надстройки (npm install/build/test) — пометь как уточняемые.
   - Раздел "Рабочий процесс": перед несвязанной задачей делать /clear; для ревью и
     исследования использовать subagents; всегда писать тесты к новому коду; сверяться с docs/.
   - Раздел "Compact-инструкции": при сжатии контекста ОБЯЗАТЕЛЬНО сохранять архитектурные
     инварианты, контракт API, причины изменений и список затронутых файлов.
   - Держи CLAUDE.md компактным и без воды — он грузится в каждую сессию.

B) Skills в .claude/skills/<name>/SKILL.md (каждый — отдельная папка с файлом SKILL.md,
   с YAML-фронтматтером name + description, где description чётко описывает, КОГДА скилл
   применять). Создай заготовки (структура + ключевые правила, без избыточного объёма):
   - clickhouse-sql: генерация безопасного агрегирующего SQL под ClickHouse; параметризация;
     контроль кардинальности; ROLLUP/GROUPING SETS для корректных итогов (инварианты 1,5,6,7).
   - pivot-shaping: укладка компактного предагрегата в оси/ячейки pivot; правила аддитивных vs
     неаддитивных итогов; детерминированность формовки (инварианты 2,7).
   - api-contract: структура запрос/ответ/метаданные/ошибки; правила версионирования
     (инвариант 8); ссылка на docs/03 и docs/contract/.
   - rust-engine-conventions: стиль Rust, обработка ошибок, async и пул соединений,
     stateless-правила, таймауты/ретраи/отмена (инвариант 2).
   - officejs-addin: ограничения Office.js, тонкий клиент, отрисовка pivot как нативной сводной
     либо форматированного диапазона (инвариант 4).
   - testing: golden SQL-тесты, юнит/интеграционные тесты, ОБЯЗАТЕЛЬНЫЕ тесты корректности
     неаддитивных итогов (инвариант 7) и контроля кардинальности (инвариант 6).
   В каждом SKILL.md в конце добавь строку-напоминание: "Не нарушай архитектурные инварианты
   из CLAUDE.md."

C) Subagents в .claude/agents/<name>.md (YAML-фронтматтер: name, description, tools; тело —
   системный промпт роли). Создай:
   - code-reviewer: старший ревьюер. Проверяет качество, безопасность и СОБЛЮДЕНИЕ инвариантов
     из CLAUDE.md. Инструменты только на чтение/поиск (Read, Grep, Glob). Возвращает конкретный
     actionable-фидбек.
   - invariant-checker: узкий проверяющий. Единственная задача — найти в изменениях нарушения
     инвариантов 1 (агрегация в Rust), 6 (контроль кардинальности), 7 (неаддитивные итоги),
     5 (инъекции/доступ вне модели). Если нарушений нет — сказать прямо. Только чтение/поиск.
   - researcher: исследователь. Изучает поведение (например, функции ClickHouse, поведение
     Office.js) в изолированном контексте и возвращает краткий вывод, не засоряя основной чат.

D) docs/contract/: создай README.md-заглушку, объясняющую, что здесь будет жить
   версионируемая JSON-схема контракта (из docs/03), и что схема — single source of truth для
   движка и надстройки одновременно.

E) .claude/settings.json (если уместно): добавь разумный permissions allowlist для частых
   безопасных команд (сборка, тесты, линт), не давая широкого доступа. Если не уверен в формате —
   создай минимальный валидный файл и пометь места для уточнения комментарием.

ПОСЛЕ СОЗДАНИЯ:
- Выведи дерево созданных файлов.
- Кратко (по одному предложению на файл) опиши назначение каждого.
- НЕ начинай реализацию Фазы 1 — дождись отдельной команды.
- Если каких-то решений не хватает (версия ClickHouse, тулчейн надстройки) — перечисли их
  как вопросы, а не выдумывай.
```

## Как пользоваться дальше

После того как Claude Code создаст окружение и вы его проверите, запускайте фазы из
`04_agentic_dev_plan.md` по одной, например:

> «Приступай к Фазе 1 из docs/04. Реализуй валидатор семантической модели и входного
> запроса по контракту docs/03. Сверяйся с FR-SEM-01..07 и NFR-SEC-07. Перед началом
> предложи план, затем реализуй с тестами. По завершении прогони `invariant-checker`.»

Перед каждой новой несвязанной задачей — `/clear`. Высокорисковые фазы (особенно
Фаза 4 — неаддитивные итоги) проверяйте вручную дополнительно к subagent-ревью.
