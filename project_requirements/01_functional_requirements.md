# Документ 01. Функциональные требования

**Проект:** OLAP-анализ данных ClickHouse в Excel через Rust-движок
**Статус:** Черновик v0.2
**Связанные документы:** `00_overview_vision.md`, `02_nonfunctional_requirements.md`, `03_architecture_and_protocol.md`

---

## 1. Соглашения

- **MUST / ДОЛЖЕН**, **SHOULD / СЛЕДУЕТ**, **MAY / МОЖЕТ** — уровни обязательности.
- Идентификаторы: `FR-<область>-<номер>`.
- Области: `ADDIN` (надстройка Office.js), `SEM` (семантический слой),
  `QUERY` (формирование SQL), `ENGINE` (Rust-движок-формовщик), `PIVOT` (сводные),
  `RESULT` (результат и рендеринг), `CACHE` (кэш), `DEPLOY` (режимы запуска),
  `MON` (наблюдаемость).

> Базовый принцип (D1/D2/D4): тяжёлая агрегация — в ClickHouse; Rust укладывает
> компактный предагрегат в форму pivot; клиент только формирует запрос и рисует.

---

## 2. Excel-надстройка / тонкий клиент (ADDIN)

### Платформа
- **FR-ADDIN-00 (MUST):** Надстройка ДОЛЖНА быть реализована на Office.js и работать в MS Excel на Windows, macOS и web из одной кодовой базы. LibreOffice/OpenOffice — вне текущей фазы (D5).

### Подключение и сессия
- **FR-ADDIN-01 (MUST):** Надстройка ДОЛЖНА позволять задать endpoint Rust-движка и проверить доступность (health-check), одинаково для локального и серверного режимов (D3).
- **FR-ADDIN-02 (MUST):** Надстройка ДОЛЖНА сохранять параметры подключения между сессиями (в пределах допустимого для Office.js, например roaming settings).
- **FR-ADDIN-03 (SHOULD):** Надстройка СЛЕДУЕТ показывать статус соединения, режим (локальный/серверный) и выбранную модель.

### Выбор семантической модели
- **FR-ADDIN-04 (MUST):** Надстройка ДОЛЖНА получать от движка список доступных моделей и отображать их.
- **FR-ADDIN-05 (MUST):** Для модели надстройка ДОЛЖНА показывать доступные измерения, меры и фильтруемые поля с их типами (получая метаданные от движка, не вычисляя ничего сама).

### Конструктор pivot (формирование запроса)
- **FR-ADDIN-06 (MUST):** Пользователь ДОЛЖЕН размещать поля в зонах rows / columns / values / filters (привычный Excel-UX).
- **FR-ADDIN-07 (MUST):** Для каждой меры пользователь ДОЛЖЕН выбирать агрегацию из разрешённых моделью (см. FR-PIVOT-02).
- **FR-ADDIN-08 (SHOULD):** Надстройка СЛЕДУЕТ поддерживать переупорядочивание измерений в зонах rows/columns.
- **FR-ADDIN-09 (MUST):** Надстройка ДОЛЖНА позволять задавать фильтры (множественный выбор, диапазоны дат/чисел).
- **FR-ADDIN-10 (SHOULD):** Надстройка СЛЕДУЕТ передавать сортировку и Top-N в запрос, чтобы они применялись в ClickHouse (а не локально).
- **FR-ADDIN-11 (MUST):** Вся работа конструктора сводится к формированию структурированного запроса (doc 03). Клиент НЕ агрегирует и НЕ пересчитывает данные.

### Выполнение и отрисовка
- **FR-ADDIN-12 (MUST):** По команде «Обновить» надстройка ДОЛЖНА отправить запрос движку и получить компактный результат.
- **FR-ADDIN-13 (MUST):** Надстройка ДОЛЖНА показывать индикатор выполнения и давать возможность отмены долгого запроса.
- **FR-ADDIN-14 (MUST):** Полученный результат ДОЛЖЕН отрисовываться на листе как сводная таблица либо эквивалентный форматированный диапазон (зависит от ограничений Office.js, R1 в doc 00). Отрисовка — единственная вычислительная задача клиента, и она НЕ пересчитывает значения, а раскладывает готовые.
- **FR-ADDIN-15 (SHOULD):** Надстройка СЛЕДУЕТ помечать результат как «устаревший», если параметры изменены, но обновление не выполнено.
- **FR-ADDIN-16 (MUST):** При ошибке надстройка ДОЛЖНА показывать понятное сообщение (без технических деталей подключения) и код/`request_id` для поддержки.

### Drill-down / интерактив
- **FR-ADDIN-17 (SHOULD):** Пользователь СЛЕДУЕТ иметь возможность expand/collapse уровней измерений, что инициирует новый запрос/довыборку у движка (а не локальный пересчёт).
- **FR-ADDIN-18 (MAY):** Надстройка МОЖЕТ поддерживать drill-through (детализирующие строки по ячейке) с жёстким лимитом строк (см. NFR).

## 3. Семантический слой (SEM)

- **FR-SEM-01 (MUST):** Система ДОЛЖНА поддерживать декларативное описание модели: таблицы ClickHouse, связи, измерения, меры.
- **FR-SEM-02 (MUST):** Для меры ДОЛЖНЫ задаваться: исходное поле/выражение и разрешённые агрегации. Все агрегации выполняются в ClickHouse (D1).
- **FR-SEM-03 (MUST):** Для измерения ДОЛЖЕН задаваться тип: категориальное, дата/время с иерархией (год→квартал→месяц→день), числовой бакет.
- **FR-SEM-04 (MUST):** Модель ДОЛЖНА задавать ограничения, гарантирующие компактность предагрегата (R2 в doc 00): максимально допустимое число групп / запрет GROUP BY по полям с высокой кардинальностью без агрегации/лимита.
- **FR-SEM-05 (SHOULD):** Модель СЛЕДУЕТ поддерживать вычисляемые меры (выражения над агрегатами), вычисляемые в Rust ПОСЛЕ получения агрегатов из ClickHouse.
- **FR-SEM-06 (MUST):** Конфигурация модели ДОЛЖНА храниться декларативно и версионироваться.
- **FR-SEM-07 (SHOULD):** Система СЛЕДУЕТ валидировать модель при загрузке (существование таблиц/полей, типы) и сообщать об ошибках.

## 4. Формирование SQL (QUERY)

- **FR-QUERY-01 (MUST):** Движок ДОЛЖЕН принимать структурированный запрос (не сырой SQL) и транслировать его в **агрегирующий** SQL к ClickHouse (тяжесть остаётся в БД, D1).
- **FR-QUERY-02 (MUST):** Сгенерированный SQL ДОЛЖЕН выполнять GROUP BY, фильтрацию, сортировку и Top-N на стороне ClickHouse, чтобы наружу выходил компактный предагрегат.
- **FR-QUERY-03 (MUST):** Все недоверенные значения ДОЛЖНЫ параметризоваться/экранироваться. Конкатенация недоверенных значений в SQL ЗАПРЕЩЕНА.
- **FR-QUERY-04 (MUST):** Движок ДОЛЖЕН использовать только поля/таблицы семантической модели; произвольные таблицы ЗАПРЕЩЕНЫ.
- **FR-QUERY-05 (MUST):** Движок ДОЛЖЕН оценивать ожидаемую кардинальность результата и отклонять запросы, способные дать некомпактный предагрегат (FR-SEM-04, R2), управляемой ошибкой ДО исполнения тяжёлого запроса.
- **FR-QUERY-06 (SHOULD):** Для итогов и неаддитивных мер движок СЛЕДУЕТ генерировать корректные агрегирующие конструкции (например `WITH ROLLUP` / `GROUPING SETS` или отдельные запросы по уровням), см. FR-PIVOT-04.
- **FR-QUERY-07 (MUST):** Движок ДОЛЖЕН логировать (debug) сгенерированный SQL с маскированием чувствительных значений.

---

## 5. Rust-движок-формовщик: исполнение (ENGINE)

- **FR-ENGINE-01 (MUST):** Движок ДОЛЖЕН быть **stateless** относительно пользовательских сессий: запрос самодостаточен, состояние между запросами не требуется (кроме кэша). Это обеспечивает горизонтальное масштабирование (D3).
- **FR-ENGINE-02 (MUST):** Движок ДОЛЖЕН исполнять SQL к ClickHouse через пул соединений.
- **FR-ENGINE-03 (MUST):** Движок ДОЛЖЕН поддерживать параллельное исполнение независимых подзапросов (например, разных уровней итогов) с настраиваемым ограничением параллелизма.
- **FR-ENGINE-04 (MUST):** Движок ДОЛЖЕН применять таймауты на подзапрос и на запрос в целом.
- **FR-ENGINE-05 (MUST):** Движок ДОЛЖЕН поддерживать отмену запроса (FR-ADDIN-13) с прерыванием обращений к ClickHouse.
- **FR-ENGINE-06 (SHOULD):** Движок СЛЕДУЕТ применять ретраи с backoff для транзиентных ошибок ClickHouse, но НЕ для ошибок запроса (синтаксис/лимиты).
- **FR-ENGINE-07 (MUST):** Поскольку наружу из ClickHouse идёт уже компактный предагрегат, движок ДОЛЖЕН удерживать малое потребление памяти на запрос; защитный лимит на размер принимаемого от ClickHouse набора ОБЯЗАТЕЛЕН (страховка от ошибки модели — FR-SEM-04/FR-QUERY-05).

## 6. Pivot и формовка (PIVOT)

- **FR-PIVOT-01 (MUST):** Движок ДОЛЖЕН укладывать предагрегат в многоуровневый pivot: несколько измерений в строках и столбцах одновременно.
- **FR-PIVOT-02 (MUST):** Поддерживаемые агрегации (вычисляются в ClickHouse): `sum`, `count`, `count distinct`, `min`, `max`, `avg`.
- **FR-PIVOT-03 (SHOULD):** Продвинутые агрегации (перцентили/квантили, медиана, stddev) СЛЕДУЕТ поддерживать; они вычисляются в ClickHouse (где это корректно при распределённых данных).
- **FR-PIVOT-04 (MUST):** Движок ДОЛЖЕН обеспечивать промежуточные и общие итоги (subtotals/grand totals) по строкам и столбцам:
  - **FR-PIVOT-04a (MUST):** Аддитивные меры (sum, count) — итоги МОГУТ собираться из частичных результатов в Rust.
  - **FR-PIVOT-04b (MUST):** Неаддитивные меры (count distinct, перцентили, avg) — итоги ДОЛЖНЫ браться из ClickHouse на соответствующем уровне агрегации (через ROLLUP/GROUPING SETS или отдельные запросы), а НЕ досчитываться суммированием подытогов в Rust (R5). Способ ДОЛЖЕН быть детерминированным и зафиксированным per-мера.
- **FR-PIVOT-05 (MUST):** Движок ДОЛЖЕН поддерживать иерархии дат (год→квартал→месяц→день) и сворачивание/разворачивание по ним (через соответствующие GROUP BY к ClickHouse при drill).
- **FR-PIVOT-06 (SHOULD):** Движок СЛЕДУЕТ поддерживать вычисляемые меры поверх агрегатов (доля от итога, разность, отношение), вычисляемые в Rust после получения агрегатов (FR-SEM-05).
- **FR-PIVOT-07 (MAY):** Движок МОЖЕТ поддерживать оконные расчёты (running total) — предпочтительно в ClickHouse, иначе в Rust на компактном наборе.
- **FR-PIVOT-08 (MUST):** Формовка ДОЛЖНА быть детерминированной: при одинаковом предагрегате результат укладки не зависит от порядка прихода частичных ответов.

---

## 7. Результат и рендеринг (RESULT)

- **FR-RESULT-01 (MUST):** Движок ДОЛЖЕН возвращать компактный структурированный результат (doc 03), готовый к отрисовке без вычислений на клиенте.
- **FR-RESULT-02 (MUST):** Результат ДОЛЖЕН включать: оси (упорядоченные значения измерений строк и столбцов), значения мер по ячейкам, метаданные (типы, форматы, признаки итога).
- **FR-RESULT-03 (MUST):** Если результат превышает безопасный лимит Excel/листа, движок ДОЛЖЕН вернуть управляемую ошибку с предложением сузить запрос — БЕЗ молчаливой обрезки.
- **FR-RESULT-04 (SHOULD):** Движок СЛЕДУЕТ возвращать признаки `from_cache` и «возраст»/`data_version` результата.
- **FR-RESULT-05 (SHOULD):** Числовые форматы (валюта, проценты, разрядность) СЛЕДУЕТ передавать как метаданные для единообразного применения клиентом.

---

## 8. Кэширование (CACHE)

- **FR-CACHE-01 (SHOULD):** Движок СЛЕДУЕТ кэшировать результаты/предагрегаты по ключу (модель + измерения + меры + фильтры + `data_version` [+ контекст прав доступа, см. R4]).
- **FR-CACHE-02 (MUST в серверном режиме):** В серверном/кластерном режиме кэш СЛЕДУЕТ делать общим (внешний кэш), чтобы не дублировать нагрузку на ClickHouse между пользователями — при условии корректного учёта прав доступа (R4).
- **FR-CACHE-03 (MUST, если кэш включён):** ДОЛЖНА быть политика инвалидации (TTL и/или маркер свежести данных ClickHouse). Согласовать (R3).
- **FR-CACHE-04 (MUST при общем кэше):** Кэш НЕ должен отдавать пользователю данные, на которые у него нет прав (актуально при пробросе identity, R4). Ключ кэша ДОЛЖЕН учитывать контекст доступа.
- **FR-CACHE-05 (SHOULD):** Пользователь СЛЕДУЕТ иметь возможность принудительно обойти кэш.

---

## 9. Режимы развёртывания (DEPLOY)

- **FR-DEPLOY-01 (MUST):** Один и тот же движок ДОЛЖЕН запускаться в двух режимах: локальном (на машине пользователя) и серверном/кластерном — без изменения контракта (D3).
- **FR-DEPLOY-02 (MUST):** Выбор режима ДОЛЖЕН определяться конфигурацией/endpoint в надстройке (FR-ADDIN-01), прозрачно для остальной логики.
- **FR-DEPLOY-03 (SHOULD):** Локальный режим СЛЕДУЕТ ориентировать на лёгкие преобразования; серверный — на большой многопользовательский масштаб с общим кэшем (FR-CACHE-02) и ограничением нагрузки на ClickHouse (см. NFR-SCALE).

---

## 10. Наблюдаемость (MON)

- **FR-MON-01 (MUST):** Движок ДОЛЖЕН логировать каждый запрос с `request_id`, длительностью, числом подзапросов к ClickHouse, размером предагрегата и результата.
- **FR-MON-02 (SHOULD):** Движок СЛЕДУЕТ экспонировать метрики (RPS, латентность по перцентилям, ошибки, кэш-хиты, объём данных из ClickHouse).
- **FR-MON-03 (MUST):** Ошибки логируются с контекстом для диагностики, но без чувствительных данных (значения фильтров маскируются).
- **FR-MON-04 (SHOULD):** `request_id` СЛЕДУЕТ возвращать клиенту для обращений в поддержку.

## 11. Трассируемость и MVP

Каждое FR покрывается хотя бы одним приёмочным тестом (матрица ведётся отдельно).
Кандидаты в MVP: FR-ADDIN-00..14, FR-SEM-01..04, FR-QUERY-01..05,
FR-ENGINE-01..05/07, FR-PIVOT-01..05, FR-RESULT-01..03, FR-DEPLOY-01..02.
Высокорисковые (обязательны к раннему покрытию тестами): FR-PIVOT-04b,
FR-QUERY-05, FR-SEM-04.
