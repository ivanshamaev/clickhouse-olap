## 2. Ключевые архитектурные решения (зафиксированы)

В отличие от ранней версии, следующие решения **подтверждены заказчиком** и больше
не являются предположениями:

| # | Решение | Следствие |
|---|---------|-----------|
| D1 | **Тяжёлая агрегация — в ClickHouse.** Сырьё (сотни млн строк) агрегируется в ClickHouse; наружу отдаётся компактный **предагрегированный** набор. | Rust НЕ тянет сырьё и НЕ делает основную агрегацию. ClickHouse используется по назначению. |
| D2 | **Rust-движок — stateless «формовщик».** Принимает ограниченный предагрегат от ClickHouse и укладывает его в форму pivot (оси строк/столбцов, ячейки, итоги) + лёгкая постобработка (вычисляемые меры, форматирование, раскладка). | Низкие требования к RAM/CPU движка на запрос. Движок легко масштабируется и может работать как локально, так и серверно. |
| D3 | **Два режима развёртывания движка из одной кодовой базы:** локальный (на машине пользователя, для лёгких преобразований) и серверный/кластерный (для большого многопользовательского масштаба). | Один бинарник, разная топология запуска. Контракт идентичен в обоих режимах. |
| D4 | **Тонкий клиент.** Excel-надстройка делает только две вещи: формирует структурированный запрос к Rust и отрисовывает полученный результат. Никаких вычислений на клиенте. | Минимум логики в надстройке; вся логика — в движке. |
| D5 | **Клиент — Office.js, только MS Office (текущая фаза.)** Покрытие: MS Excel на Windows / macOS / web одной кодовой базой. | LibreOffice/OpenOffice — отдельная будущая фаза (см. ниже). |

### Осознанно отложено (будущие фазы)
- **LibreOffice / OpenOffice клиент.** Office.js физически не работает в LibreOffice/OpenOffice (у них иная модель расширений — UNO). Поскольку клиент тонкий, в будущем можно добавить отдельный UNO-клиент **поверх того же контракта к Rust**, не меняя движок. На текущей фазе — out of scope.

## 3. Проблема и бизнес-цель

### 3.1. Текущая ситуация
Аналитические данные (сотни млн строк) хранятся в ClickHouse — колоночной СУБД,
оптимизированной под быструю агрегацию больших объёмов. Бизнес-пользователи работают
в Excel и привыкли к сводным таблицам (pivot) как к основному инструменту анализа.

Прямое подключение Excel к ClickHouse (например, generic ODBC) ограничено:
неоптимальная генерация SQL, риск выгрузить в Excel объём сверх его лимита
(~1 048 576 строк на лист), слабая поддержка многоуровневых pivot.

### 3.2. Цель
Дать пользователю привычный pivot-опыт в Excel поверх ClickHouse так, чтобы:
- тяжёлая агрегация оставалась в ClickHouse (D1);
- по сети и в Excel передавался только компактный предагрегат (D2);
- Rust быстро укладывал предагрегат в форму pivot;
- интерактив (drill-down, смена измерений/мер) был отзывчивым.

### 3.3. Метрики успеха (черновые)
- Время отклика типового pivot-запроса (детально — в `02_*`).
- Объём предагрегата, передаваемого в Excel, гарантированно в пределах лимита листа.
- Доля интерактивных операций, обслуженных без полного пересчёта (кэш/довыборка).

## 4. Участники и роли

| Роль | Описание | Интересы |
|------|----------|----------|
| Бизнес-аналитик | Конечный пользователь надстройки | Скорость, привычный pivot-UX, корректность чисел |

| Разработчик движка / клиента | Команды реализации | Чёткий стабильный контракт |

## 5. Границы решения (Scope)

### 5.1. В составе (In scope, текущая фаза)
1. Тонкая Excel-надстройка на **Office.js (только MS Office)**: конструктор pivot + отрисовка.
2. **Rust-движок-формовщик** (stateless): приём запроса → генерация SQL к ClickHouse (агрегирующего) → приём предагрегата → укладка в форму pivot → возврат компактного результата.
3. Контракт обмена «надстройка ↔ движок» (см. doc 03).
4. Слой подключения к ClickHouse (пул, ретраи, таймауты).
5. Семантический слой (метаданные: таблицы, поля, меры, типы агрегаций).
6. Кэширование результатов/предагрегатов (особенно важно в серверном режиме).
7. Оба режима развёртывания движка (локальный и серверный/кластер).

### 5.2. Вне состава (Out of scope, текущая фаза)
1. Клиент под LibreOffice/OpenOffice (будущая фаза, отдельный UNO-клиент).
2. ETL/загрузка данных в ClickHouse (данные уже там).
3. Управление кластером ClickHouse.
4. BI-дашборды вне Excel.
5. Запись данных обратно в ClickHouse (write-back).
6. ML/прогнозная аналитика.
7. Тяжёлая агрегация сырья на стороне Rust (по D1/D2 — принципиально не делается).

## 6. Высокоуровневый поток

```
Пользователь в Excel (MS Office)
        │ выбирает измерения / меры / фильтры
        ▼
Тонкая Office.js-надстройка ──(структурированный запрос, doc 03)──► Rust-движок
                                                                       │
                          генерация АГРЕГИРУЮЩИХ SQL (тяжесть — в БД)   │
                                                                       ▼
                                                                 ClickHouse
                                                                       │
                          КОМПАКТНЫЙ предагрегат (ограниченный объём)   │
                                                                       ▼
                          Rust: укладка в форму pivot + лёгкая          │
                                постобработка (оси, ячейки, итоги)      │
                                                                       │
        ◄──(компактный pivot-результат)─────────────────────────────────┘
Надстройка ОТРИСОВЫВАЕТ результат на лист (вычислений на клиенте нет)
```

Деплой: тот же движок запускается либо локально (рядом с Excel), либо как
серверный/кластерный сервис. Топология не меняет контракт.

## 7. Риски и открытые вопросы

| # | Риск / вопрос | Комментарий |
|---|---------------|-------------|
| R1 | Office.js ограничивает доступ к нативным PivotTable-объектам Excel | Возможно, отрисовка как форматированный диапазон («псевдо-pivot»), а не нативная сводная. Проверить на целевых версиях MS Office (Win/Mac/web). |
| R2 | Гарантия «компактности» предагрегата | Семантическая модель и движок ДОЛЖНЫ не допускать запросов, дающих гигантский «предагрегат» (например, GROUP BY по высоко-кардинальному полю). Нужны лимиты по кардинальности/числу групп. |
| R3 | Гранулярность кэша vs. свежесть данных | Особенно важно в серверном многопользовательском режиме. Политику инвалидации согласовать. |
| R4 | Аутентификация до ClickHouse: проброс identity пользователя vs. сервисная учётка | Влияет на row-level доступ и на кэш (общий кэш при разных правах доступа — риск утечки). Открытый вопрос. |
| R5 | Корректность неаддитивных итогов (count distinct, перцентили, avg) | Их нельзя получить суммированием подытогов. Раз агрегирует ClickHouse (D1), итоговые уровни ДОЛЖНЫ запрашиваться у ClickHouse корректно, а не досчитываться в Rust суммированием. |
| R6 | Локальный режим и доступ к ClickHouse с машины пользователя | Сетевой доступ, учётные данные, файрволы. В серверном режиме проще; в локальном — согласовать с IT. |
